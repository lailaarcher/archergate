How to Add Licensing to a Desktop App

April 2026 / 7 min read

The landscape of desktop app licensing.

Desktop applications have a distribution problem that web apps do not. When your software runs on the user's machine, the user has the binary. They can copy it, share it, reverse-engineer it, or redistribute it. Whether you are building with Electron, Tauri, Qt, WPF, or a native C++ toolkit, you face the same fundamental challenge: how do you verify that the person running your software actually paid for it?

The answer involves three components: a license key (something the user receives at purchase), a machine fingerprint (something that identifies the user's hardware), and a validation server (something that binds the key to the fingerprint). This article walks through how to implement all three for any desktop framework.

Machine fingerprinting.

Machine fingerprinting is the process of computing a stable, unique identifier for a specific computer. The fingerprint should not change when the user updates their OS or installs new software, but it should change when they move to a different machine.

In practice, a good fingerprint combines two signals: the CPU brand string and the OS installation identifier. The CPU brand string is stable across reboots and OS updates. The OS installation ID changes when the user reinstalls the OS, which is an acceptable re-activation trigger. Combining both prevents fingerprint collisions (two identical laptops with fresh OS installs would have the same OS ID but different CPU serial numbers are not reliably available, so the brand string provides differentiation when combined with other factors).

The Archergate SDK computes this fingerprint internally, but if you are integrating via the REST API, you need to compute it yourself. Here is how to obtain the OS installation identifier on each platform.

Platform Source Access Method
Windows MachineGuid Registry: HKLM\SOFTWARE\Microsoft\Cryptography
macOS IOPlatformUUID ioreg -rd1 -c IOPlatformExpertDevice
Linux /etc/machine-id Read file contents directly

Concatenate the CPU brand string with the OS ID, then SHA-256 hash the result. This gives you a 64-character hex string that uniquely identifies the machine. Send this as the machine_fingerprint field in your API call.

Integration for Electron apps.

Electron apps run on Node.js. You cannot link a static C library directly, but you can call the Archergate REST API from your main process. The validation call should happen in your main process (not the renderer) to prevent users from bypassing it through DevTools.

main.js (Electron main process)
const { execSync } = require('child_process'); const crypto = require('crypto'); const os = require('os'); function getMachineFingerprint() { let osId; if (process.platform === 'win32') { osId = execSync( 'reg query HKLM\\\\SOFTWARE\\\\Microsoft\\\\Cryptography /v MachineGuid' ).toString().match(/REG_SZ\s+(.+)/)[1].trim(); } else if (process.platform === 'darwin') { osId = execSync( 'ioreg -rd1 -c IOPlatformExpertDevice' ).toString().match(/IOPlatformUUID.*?"(.+?)"/)[1]; } else { osId = require('fs').readFileSync( '/etc/machine-id', 'utf8' ).trim(); } const cpuBrand = os.cpus()[0].model; return crypto.createHash('sha256') .update(cpuBrand + osId).digest('hex'); } async function validateLicense(licenseKey) { const resp = await fetch( 'https://your-server.com/validate', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ license_key: licenseKey, machine_fingerprint: getMachineFingerprint(), plugin_id: 'com.yourname.yourapp' }) }); return resp.json(); }

Store the validation result in a signed local file so the app works offline. Re-validate on a schedule (every 7 or 30 days) rather than on every launch.

Integration for Tauri apps.

Tauri apps use a Rust backend, which means you can link the Archergate SDK directly as a static library. This is the most secure option because the validation logic is compiled into native code, not sitting in JavaScript where it can be patched.

Cargo.toml
[dependencies] archergate-license = "0.1"
src-tauri/src/main.rs
use archergate_license::LicenseClient; #[tauri::command] fn check_license(key: String) -> Result<String, String> { let client = LicenseClient::new( "YOUR_API_KEY", "com.yourname.app" ); match client.validate(&key) { Ok(status) => Ok(format!("{:?}", status)), Err(e) => Err(e.to_string()), } }

Because the validation runs in compiled Rust on the backend, a user cannot bypass it by modifying frontend JavaScript. The Tauri IPC bridge only receives the result, not the validation logic.

Integration for Qt and native C++ apps.

Qt applications and other native C++ apps can use the Archergate C++ wrapper directly. Link the static library and include the header.

main.cpp
#include "archergate_license.h" int main(int argc, char* argv[]) { archergate::License license( "YOUR_API_KEY", "com.yourname.qtapp" ); auto result = license.validate(storedLicenseKey); if (result.status != archergate::Status::Valid) { // Show activation dialog showActivationWindow(); return 1; } // Proceed with application QApplication app(argc, argv); // ... }

REST API for any language.

If your app is written in Python, C#, Go, Java, or any other language, you can call the Archergate validation server over HTTP. The API is a single POST endpoint.

HTTP request
POST /validate HTTP/1.1 Host: your-license-server.com Content-Type: application/json { "license_key": "XXXX-XXXX-XXXX-XXXX", "machine_fingerprint": "a1b2c3d4...", "plugin_id": "com.yourname.app" }
Response (valid)
{ "valid": true, "status": "active", "expires_at": "2027-04-03T00:00:00Z", "machine_bound": true }

Compute the machine fingerprint on the client side (as shown in the Electron example above), send it with the license key, and cache the server's response locally. The server handles key validation, machine binding enforcement, and activation counting. Your app just needs to make one HTTP call and store the result.

Offline validation and caching.

Users will not always have internet access when they launch your application. The standard pattern is to cache the last successful validation response on disk, signed with an HMAC key that you embed in your binary. On subsequent launches, if the server is unreachable, the app checks the local cache. If the cache is less than 30 days old and the HMAC signature is valid, the app proceeds normally.

The Archergate SDK handles this automatically for C, C++, and Rust apps. For REST API integrations, you implement caching on your side. The important properties: the cache file must be signed (so users cannot edit it), it must include the machine fingerprint (so users cannot copy it between machines), and it must include a timestamp (so it expires).

Server deployment.

The Archergate license server is a single Rust binary. It uses SQLite for storage, so there is no separate database to manage. Deploy it on a $5/month VPS, a Docker container, a Fly.io instance, or any server that can listen on a port. Generate license keys through the admin API, wire your payment system (Stripe webhook, Gumroad ping, manual CSV) to create keys on purchase, and your infrastructure is complete.
