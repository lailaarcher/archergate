Machine-Bound Licensing Explained

April 2026 / 8 min read

What is machine-bound licensing?

Machine-bound licensing (also called node-locked licensing or hardware-locked licensing) ties a software license key to a specific physical computer. When a user activates their license, the software computes a fingerprint of the machine's hardware and OS, then registers that fingerprint with a license server. On subsequent launches, the software recomputes the fingerprint and checks it against the registered value. If the fingerprints do not match, the license is invalid.

This prevents the simplest and most common form of software piracy: copying. A user cannot buy one license, then share the binary and key with colleagues. The key only works on the machine it was first activated on. The user experience is minimal friction: enter a key, click activate, and the software works. No dongles, no login accounts, no recurring authentication prompts.

How hardware fingerprinting works.

A machine fingerprint is a deterministic function of hardware and OS properties. The function must satisfy three constraints:

  1. Stability: The fingerprint should not change across reboots, software updates, or typical system maintenance. Users should not need to re-activate after installing a Windows update.
  2. Uniqueness: Two different machines should produce different fingerprints. If two users have the same fingerprint, the system cannot distinguish their activations.
  3. Non-transferability: The fingerprint should be difficult to spoof. A user should not be able to set a registry value or environment variable to impersonate another machine's fingerprint.

No single hardware property satisfies all three constraints perfectly. CPU serial numbers (CPUID) are not reliably unique across consumer processors. MAC addresses can be spoofed. Disk serial numbers change when a drive is replaced. The practical solution is to combine multiple signals into a composite fingerprint and accept that the result is probabilistically unique rather than cryptographically guaranteed.

The Archergate fingerprinting approach

Archergate computes a SHA-256 hash of two concatenated values: the CPU brand string and the OS installation identifier. Here is how each component is obtained on each platform.

Windows

The CPU brand string is read via the CPUID instruction (returned by the OS through standard library calls). The OS installation identifier is the MachineGuid value stored in the Windows registry at HKLM\SOFTWARE\Microsoft\Cryptography. This GUID is generated during Windows installation and persists across updates. It changes only if the user reinstalls Windows from scratch.

Windows fingerprint source
Registry: HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Cryptography Value: MachineGuid Type: REG_SZ Example: d85b3a32-7c6e-4c90-b8f3-9b1234abcdef

macOS

The CPU brand string is obtained from sysctl. The OS installation identifier is the IOPlatformUUID, a hardware UUID exposed by the IOKit framework. On Macs, this UUID is stored in firmware and is unique to each machine. It persists across macOS reinstalls, which makes it more stable than the Windows equivalent.

macOS fingerprint source
$ ioreg -rd1 -c IOPlatformExpertDevice | grep IOPlatformUUID "IOPlatformUUID" = "A1B2C3D4-E5F6-7890-ABCD-EF1234567890"

Linux

The CPU brand string comes from /proc/cpuinfo. The OS installation identifier is the contents of /etc/machine-id, a 32-character hex string generated during system installation by systemd. Most modern Linux distributions (Ubuntu, Fedora, Arch, Debian) generate this file during initial setup. It persists across package updates and kernel upgrades, but changes on a fresh install.

Linux fingerprint source
$ cat /etc/machine-id a1b2c3d4e5f6789012345678abcdef01

Computing the final fingerprint

The two values are concatenated and hashed:

fingerprint = SHA256(cpu_brand_string + os_installation_id)

This produces a 64-character hexadecimal string. The SHA-256 hash serves two purposes: it normalizes the length of the fingerprint regardless of input length, and it makes it computationally difficult to reverse-engineer the original inputs from the fingerprint (useful if the fingerprint is transmitted over the network and intercepted).

Security properties.

Machine-bound licensing provides specific, well-defined security guarantees. It is important to understand what it does and does not protect against.

Prevents casual copying. If a user emails the binary and license key to a friend, the friend's machine will produce a different fingerprint. The key will not activate on the second machine. This is the primary threat model for indie software: users sharing with friends, coworkers, or posting keys online.

Prevents key sharing services. Sites that distribute license keys are less effective because each key is bound to one machine. A key posted online can only be used by the person who originally activated it.

Does not prevent binary patching. A skilled reverse engineer can patch the binary to skip the license check entirely. No licensing scheme that runs on the user's machine can fully prevent this. Machine binding raises the bar (the attacker needs reverse engineering skills, not just a copied key) but does not make it impossible.

Does not prevent VM cloning. A user could activate the license inside a virtual machine, then distribute a copy of that VM image. The fingerprint would match because the virtualized hardware is identical. This is an edge case that rarely occurs in practice for desktop applications, but it is a real limitation.

Practical perspective: No copy protection is unbreakable. The goal is not to build an impenetrable fortress. The goal is to make the path of least resistance be purchasing a license. Machine binding achieves this for the vast majority of users. The small percentage who patch binaries or distribute VM images were never going to pay regardless.

When to use machine binding.

Machine-bound licensing is the right choice when your software meets these criteria:

  • It runs as a native desktop application (not a web app or mobile app).
  • Users typically install it on one or two machines (workstation, laptop).
  • Your user base is primarily individuals or small teams, not enterprise accounts with hundreds of seats.
  • You want low-friction licensing: no dongles, no account creation, no recurring login.
  • You can tolerate a one-time internet connection for activation (with offline grace afterward).

When to choose a different approach.

Use account-based licensing if your users need to access the software on many different machines (roaming use cases). Account-based licensing ties the license to a user identity rather than a machine. The tradeoff is that users must create accounts and authenticate on each launch.

Use floating/concurrent licensing if you sell to organizations that need a pool of licenses shared across many employees. A floating license server tracks how many simultaneous users are active and enforces a seat limit. This is common for enterprise CAD software and studio tools but is overkill for indie products.

Use hardware dongles (iLok) if you are selling very high-value professional software where the cost of the dongle is trivial relative to the software price, and your users already own dongles for other software. This is common in professional audio (Pro Tools, Waves) but rarely appropriate for indie developers.

Use no protection if your software is free or open source, if your primary revenue comes from services rather than licenses, or if you have determined that the engineering cost of implementing protection exceeds the revenue lost to piracy.

Implementation with Archergate.

Archergate handles the entire machine-binding workflow: fingerprint computation, server communication, activation tracking, offline caching, and tamper detection. It ships as a static library for C, C++, and Rust, plus a REST API for other languages.

The server side is a single Rust binary backed by SQLite. It tracks which license keys are activated, which machine fingerprints they are bound to, how many activations each key has used (configurable limit), and when each activation expires. You run this server on your own infrastructure.
